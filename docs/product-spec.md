## 2. 产品背景

开发者电脑中通常存在多个通过 Git 管理的项目。这些项目可能分布在不同磁盘和不同目录中。

用户每次需要打开项目时，通常需要：

1. 打开文件资源管理器。
    
2. 找到项目所在目录。
    
3. 判断应该使用哪个 IDE。
    
4. 启动 IDE 并打开项目。
    
5. 通过终端或 IDE 查看当前分支和 Git 状态。
    

当本地项目数量逐渐增加后，这种操作方式存在以下问题：

- 项目路径难以记忆。
    
- 查找项目需要多次进入目录。
    
- 不同项目可能需要使用不同 IDE。
    
- 无法快速查看项目当前分支和本地修改状态。
    
- 项目移动或磁盘暂时不可用时，不容易发现。
    
- 缺少统一的项目入口。
    

因此，需要开发一款 Windows 桌面端软件，用于集中管理电脑中已经存在的 Git 项目。

## 3. 产品目标

### 3.1 核心目标

让用户打开软件后，可以快速完成以下操作：

- 查看电脑中已经导入的 Git 项目。
    
- 通过项目名称或路径搜索项目。
    
- 查看项目的基础 Git 状态。
    
- 使用指定 IDE 打开项目。
    
- 切换本地分支或远程分支。
    
- 主动检查远程更新。
    
- 确认后执行 Pull。
    
- 识别项目路径失效和 Git 冲突状态。
    

### 3.2 MVP 成功标准

用户可以在不打开文件资源管理器、不手动进入项目目录的情况下，快速找到并打开目标项目。

用户可以在软件中完成基础 Git 状态查看、分支切换、Fetch 和 Pull，但代码编辑、提交和 Push 仍然在 IDE 或终端中完成。

## 4. 产品边界

本产品不是完整的 Git 客户端，也不是团队协作项目管理平台。

### 4.1 MVP 包含

- 本地 Git 项目扫描和导入。
    
- 项目列表管理。
    
- 项目搜索。
    
- 项目最近打开记录。

- 查看未提交文件列表。
    
- 解决 Git 冲突。
    
- 完整 Git 操作历史。
    
- GitHub、GitLab 等账号登录。
    
- Token、密码或 SSH Key 管理。
    
- 团队协作和成员管理。
    
- 项目任务管理。
    
- 项目技术栈识别。
    
- 收藏、标签和项目分组。
    
- 云同步。
    
- 配置导入和导出。
    
- 全局快捷键。
    
- 开机自动启动。
    
- 多窗口。
    
- 多语言。
    
- 使用数据统计和崩溃自动上报。

## 5. 目标用户

### 5.1 用户画像

主要用户为个人开发者，具有以下特征：

- 使用 Windows 电脑进行开发。
    
- 本地保存多个 Git 项目。
    
- 项目分布在一个或多个工作目录中。
    
- 经常使用 IntelliJ IDEA、PyCharm、WebStorm、VS Code 等 IDE。
    
- 需要频繁在不同项目之间切换。
    
- 希望快速了解项目当前分支和修改状态。
    
- 不希望项目名称、路径或 Git 信息上传到云端。
    

### 5.2 典型使用场景

用户电脑中存在以下目录：

```
D:\Projects
E:\Workspace
D:\Learning\AI-Agent
```

这些目录中包含多个 Java、Python、前端或其他类型的 Git 项目。

用户打开本软件后，可以：

1. 搜索 `enterprise-admin`。
    
2. 点击项目进入详情。
    
3. 查看当前分支和未提交修改数量。
    
4. 点击“打开项目”。
    
5. 软件使用该项目上次选择的 IntelliJ IDEA 打开项目。

## 6. 核心用户流程

### 6.1 首次启动流程

1. 用户启动软件。
    
2. 软件展示欢迎页面。
    
3. 用户选择一个或多个扫描目录。
    
4. 软件递归扫描目录中的 Git 仓库。
    
5. 软件展示扫描结果。
    
6. 所有扫描结果默认勾选。
    
7. 嵌套仓库显示“嵌套项目”标记。
    
8. 用户取消不需要导入的项目。
    
9. 用户点击“导入项目”。
    
10. 软件检测本机可用 IDE。
    
11. 软件进入项目首页。
    

### 6.2 日常打开项目流程

1. 用户打开软件。
    
2. 软件立即展示缓存的项目列表和 Git 状态。
    
3. 软件在后台逐个刷新本地 Git 状态。
    
4. 用户通过项目名称或路径搜索项目。
    
5. 用户点击项目进入详情页。
    
6. 用户点击“打开项目”。
    
7. 如果项目此前没有选择过 IDE，软件弹出 IDE 选择窗口。
    
8. 用户选择 IDE。
    
9. 软件记住该项目选择的 IDE。
    
10. 软件启动 IDE，并更新项目最近打开时间。
    

### 6.3 Pull 流程

1. 用户进入项目详情页。
    
2. 用户点击“Pull”。
    
3. 软件检查项目路径、Git 状态和当前分支。
    
4. 软件主动检查远程状态。
    
5. 软件展示 Pull 确认窗口。
    
6. 用户确认 Pull 策略和目标分支。
    
7. 软件执行 Pull。
    
8. 软件展示成功、失败或冲突结果。
    
9. 软件保存该项目最近一次 Git 操作记录。
    

### 6.4 分支切换流程

1. 用户进入项目详情页。
    
2. 用户打开分支列表。
    
3. 用户搜索并选择目标分支。
    
4. 软件检查是否存在未提交修改。
    
5. 软件展示必要的风险提示。
    
6. 用户确认切换。
    
7. 软件执行分支切换。
    
8. 软件刷新当前分支和 Git 状态。
    
9. 软件保存最近一次切换结果。

# 8. 功能需求

## 8.1 项目扫描目录管理

### 8.1.1 添加扫描目录

用户可以添加一个或多个需要扫描的根目录，例如：

```
D:\Projects
E:\Workspace
```

软件保存用户添加的扫描目录。

### 8.1.2 删除扫描目录

用户可以从扫描目录配置中移除某个目录。

移除扫描目录不会：

- 删除硬盘中的任何文件。
    
- 自动删除已经导入的项目记录。
    

### 8.1.3 手动添加项目

除目录扫描外，用户可以手动选择一个包含 Git 仓库的项目目录。

添加前需要验证：

- 目录是否存在。
    
- 目录是否具有访问权限。
    
- 目录是否为 Git 仓库。
    
- 项目路径是否已经存在于软件中。
    

---

## 8.2 Git 项目扫描

### 8.2.1 Git 仓库识别

软件通过项目目录中的 Git 元数据识别 Git 仓库。

扫描过程不得修改项目文件。

### 8.2.2 启动检查

每次启动软件时，执行快速检查：

- 检查已配置扫描目录是否可访问。
    
- 检查是否存在新项目。
    
- 检查已导入项目路径是否仍然有效。
    
- 不阻塞首页项目列表展示。
    

### 8.2.3 手动完整刷新

用户可以点击“刷新全部”执行完整扫描。

完整刷新包括：

- 重新扫描所有扫描目录。
    
- 发现新 Git 项目。
    
- 检查项目路径状态。
    
- 更新项目基础 Git 状态。
    
- 跳过被忽略的目录。
    

### 8.2.4 扫描结果确认

扫描完成后展示：

- 项目名称。
    
- 项目完整路径。
    
- 是否属于嵌套仓库。
    
- 扫描警告。
    

所有项目默认勾选，用户可以取消不希望导入的项目。

### 8.2.5 嵌套仓库

父目录和子目录都为 Git 仓库时：

- 两个仓库都显示在扫描结果中。
    
- 子仓库显示“嵌套项目”标记。
    
- 显示其所属父仓库。
    
- 用户决定是否分别导入。
    
- 导入后作为两个独立项目管理。
    

---

## 8.3 扫描忽略规则

### 8.3.1 内置忽略目录

MVP 默认跳过以下常见目录：

```
node_modules
target
build
dist
out
vendor
.idea
.vscode
.gradle
.next
coverage
```

### 8.3.2 自定义全局忽略规则

用户可以添加全局忽略目录或路径规则，例如：

```
D:\Projects\archive
**\generated
**\test-data
```

### 8.3.3 忽略规则约束

- 忽略规则只影响后续扫描。
    
- 不修改或删除本地目录。
    
- 已经导入的项目不会因为新增忽略规则而自动移除。
    
- 用户可以恢复默认忽略规则。
    
- 修改规则后，用户可以手动执行完整刷新。
    

### 8.3.4 无权限目录

遇到无权限访问的目录时：

- 跳过该目录。
    
- 扫描任务继续执行。
    
- 扫描结果中显示警告。
    
- 在本地错误日志中记录错误。

## 8.4 项目首页

### 8.4.1 视图模式

首页支持：

- 卡片视图。
    
- 表格视图。
    

软件记住用户上次使用的视图模式。

### 8.4.2 排序方式

所有项目统一按照最近打开时间倒序排列：

- 最近成功通过 IDE 打开的项目排在最前面。
    
- 从未打开过的项目排在后面。
    
- MVP 不支持手动切换其他排序方式。
    

### 8.4.3 搜索

搜索范围包括：

- 项目自定义显示名称。
    
- 项目文件夹名称。
    
- 项目完整路径。
    

搜索结果继续按照最近打开时间排序。

### 8.4.4 项目卡片或表格字段

首页至少展示：

- 项目显示名称。
    
- 项目路径。
    
- 当前分支。
    
- 未提交修改数量。
    
- 最近打开时间。
    
- 默认 IDE。
    
- 路径状态。
    
- Git 状态刷新状态。
    

### 8.4.5 项目点击行为

- 单击项目进入项目详情页。
    
- 单击项目不会直接打开 IDE。
    
- 必须点击明确的“打开项目”按钮才启动 IDE。

## 8.5 项目名称

### 8.5.1 默认名称

项目首次导入时，使用项目文件夹名称作为显示名称。

### 8.5.2 自定义名称

用户可以修改项目在软件中的显示名称。

修改显示名称不会：

- 修改本地文件夹名称。
    
- 修改 Git 仓库名称。
    
- 修改远程仓库配置。
    

### 8.5.3 名称恢复

用户可以将显示名称恢复为项目文件夹名称。

### 8.5.4 同名项目

允许存在同名项目，通过完整路径进行区分。

### 8.6.1 基础信息区域

展示：

- 项目文件夹名称。

- 从软件中移除项目。
    

从软件中移除项目只删除软件中的记录，不删除硬盘中的项目文件。

## 8.7 项目路径状态

### 8.7.1 路径失效识别

出现以下情况时，将项目标记为“路径失效”：

- 项目目录已被删除。
    
- 项目目录已被移动。
    
- 所在磁盘未连接。
    
- 当前用户无权访问项目目录。
    

### 8.7.2 路径失效后的行为

路径失效时：

- 保留项目记录。
    
- 不自动从列表中删除。
    
- 禁用打开项目。
    
- 禁用 Git 状态刷新。
    
- 禁用 Fetch、Pull 和分支切换。
    
- 提供“重新指定路径”。
    
- 提供“从列表移除”。
    

### 8.7.3 路径恢复

原路径重新可用后，用户执行刷新，项目自动恢复正常状态。

- 允许用户重新配置或选择其他 IDE。

## 8.9 使用 IDE 打开项目

### 8.9.1 首次打开

项目第一次点击“打开项目”时：

1. 弹出可用 IDE 列表。
    
2. 用户选择一个 IDE。
    
3. 软件使用该 IDE 打开项目。
    
4. 软件将其保存为该项目默认 IDE。
    

### 8.9.2 后续打开

项目已经设置默认 IDE 时：

- 点击“打开项目”直接使用默认 IDE。
    
- 不再重复弹出选择窗口。
    

### 8.9.3 临时使用其他 IDE

用户可以通过“使用其他 IDE 打开”临时选择 IDE。

临时选择默认不修改项目默认 IDE，除非用户明确选择“设为默认”。

### 8.9.4 最近打开时间

只有 IDE 成功启动后，才更新项目最近打开时间。

IDE 启动失败时，不更新时间。

## 8.10 Git 环境

### 8.10.1 本机 Git

软件调用用户电脑中已经安装的 Git。

软件不内置 Git，也不管理 Git 版本。

### 8.10.2 Git 不可用

未检测到 Git 时：

- 项目仍可以被导入和使用 IDE 打开。
    
- Git 状态区域显示“未检测到 Git”。
    
- 禁用 Fetch、Pull 和分支切换。
    
- 提供中文安装提示。
    

### 8.10.3 身份验证

软件完全复用本机 Git 身份验证配置，包括：

- SSH Key。
    
- Git Credential Manager。
    
- 系统中已经保存的 GitHub 或 GitLab 凭据。
    
- Git 项目已有的远程仓库配置。
    

软件不得保存：

- GitHub 密码。
    
- GitLab 密码。
    
- Personal Access Token。
    
- SSH 私钥。

## 8.11 Git 状态刷新

### 8.11.1 启动展示

软件启动后：

1. 立即读取本地缓存。
    
2. 展示项目列表和上次 Git 状态。
    
3. 后台逐个刷新项目本地 Git 状态。
    
4. 不等待所有项目刷新完成后再展示首页。
    

### 8.11.2 后台刷新

后台刷新：

- 只执行本地 Git 状态检查。
    
- 不自动访问远程仓库。
    
- 限制并发数量。
    
- 不阻塞用户搜索、查看和打开项目。
    

### 8.11.3 进入详情页

进入项目详情页时：

- 当前项目获得较高刷新优先级。
    
- 立即刷新本地 Git 状态。
    
- 不自动执行 `git fetch`。
    

### 8.11.4 手动刷新

项目详情页提供“刷新状态”按钮，只刷新当前项目。

首页提供“刷新全部”按钮，用于重新扫描目录并刷新所有项目。

### 8.11.5 刷新失败

刷新失败时：

- 保留上一次成功状态。
    
- 标记“刷新失败”。
    
- 显示中文错误说明。
    
- 允许查看和复制原始错误。
    
- 不影响其他项目刷新。

## 8.12 远程状态检查

### 8.12.1 默认行为

软件后台不自动执行 `git fetch`。

首页展示的领先、落后数量可能来自最近一次 Fetch 的结果。

### 8.12.2 状态提示

远程状态需要展示：

- 最后一次远程检查时间。
    
- 是否可能已经过期。
    
- “远程状态未更新”提示。
    

### 8.12.3 主动检查

用户点击“检查远程更新”时：

1. 执行 `git fetch`。
    
2. Fetch 完成后重新计算领先和落后数量。
    
3. 刷新远程分支列表。
    
4. 保存最近一次 Git 操作结果。
    

### 8.12.4 Fetch 失败

Fetch 失败时：

- 保留原有远程状态。
    
- 显示网络、认证或远程仓库错误。
    
- 提供原始 Git 输出。
    
- 提供“在终端中打开项目”。

```
未提交修改：5 个文件
```

MVP 不支持：

- 查看具体文件列表。
    
- 查看文件状态。
    
- 查看 Diff。
    
- 暂存文件。
    
- 丢弃修改。
    
- 提交代码。

## 8.14 分支管理

### 8.14.1 分支列表

展示：

- 当前分支。
    
- 本地分支。
    
- 远程分支。
    

支持按分支名称搜索。

### 8.14.2 本地分支切换

没有未提交修改时，可以确认后切换本地分支。

### 8.14.3 远程分支切换

用户选择没有对应本地跟踪分支的远程分支时：

- 展示将要创建的本地分支名称。
    
- 展示对应远程分支。
    
- 用户确认后创建本地跟踪分支并切换。
    

### 8.14.4 存在未提交修改

存在未提交修改时：

- 软件必须显示风险提示。
    
- 不自动执行 `git stash`。
    
- 不强制覆盖本地修改。
    
- Git 不允许切换时，展示具体错误。
    
- 用户需要通过 IDE 或终端自行处理。
    

### 8.14.5 切换失败

切换失败时：

- 保持原分支状态。
    
- 显示中文说明和 Git 原始错误。
    
- 保存为最近一次 Git 操作结果。

## 8.15 Pull

### 8.15.1 Pull 策略

默认使用：

```
git pull --rebase
```

支持三种 Pull 策略：

|   |   |
|---|---|
|策略|Git 行为|
|Rebase|`git pull --rebase`|
|Merge|`git pull --no-rebase`|
|遵循 Git 配置|`git pull`|

### 8.15.2 配置层级

Pull 策略优先级：

1. 项目独立设置。
    
2. 软件全局默认设置。
    
3. 软件内置默认值 Rebase。
    

### 8.15.3 Pull 确认窗口

执行 Pull 前必须展示：

- 项目名称。
    
- 当前本地分支。
    
- 远程跟踪分支。
    
- Pull 策略。
    
- 未提交修改数量。
    
- 本地领先数量。
    
- 本地落后数量。
    
- 远程状态检查时间。
    

用户确认后才执行 Pull。

### 8.15.4 本次临时策略

用户可以在确认窗口中临时修改本次 Pull 策略。

临时修改默认不改变全局设置或项目设置。

### 8.15.5 Pull 前远程检查

执行 Pull 前主动检查远程状态。

远程检查失败时，不直接继续 Pull，应提示用户确认下一步处理。

### 8.15.6 存在未提交修改

存在未提交修改时：

- 默认阻止 Pull。
    
- 提示用户先通过 IDE 或终端处理修改。
    
- 不自动执行 `git stash`。
    
- 不自动提交代码。
    

### 8.15.7 Pull 成功

Pull 成功后：

- 刷新当前分支。
    
- 刷新最近提交信息。
    
- 刷新领先和落后数量。
    
- 保存最近一次 Git 操作记录。
    
- 展示成功提示。
    

---

## 8.16 Git 冲突处理

### 8.16.1 冲突识别

Pull 发生 Merge 或 Rebase 冲突时：

- 停止当前 Pull 流程。
    
- 将项目标记为“存在冲突”。
    
- 统计并展示冲突文件数量。
    

### 8.16.2 冲突状态操作

提供：

- 使用默认 IDE 打开项目。
    
- 使用其他 IDE 打开项目。
    
- 在终端中打开项目。
    
- 刷新 Git 状态。
    

### 8.16.3 不支持的冲突操作

软件不执行：

- 自动解决冲突。
    
- `git rebase --continue`。
    
- `git rebase --abort`。
    
- `git merge --continue`。
    
- `git merge --abort`。
    
- 冲突文件内容查看或编辑。
    

用户处理冲突后，需要手动刷新项目状态。

## 8.17 最近一次 Git 操作记录

每个项目只保留最近一次 Git 操作记录。

记录内容包括：

- 操作类型。
    
- 操作时间。
    
- 成功、失败或冲突状态。
    
- 简要结果。
    
- 简要错误原因。
    

操作类型包括：

- Fetch。
    
- Pull。
    
- 切换分支。
    

新的操作记录覆盖旧记录。

MVP 不提供完整操作历史页面。

- 在 PowerShell 或 Windows Terminal 中打开项目目录。
    

项目路径失效时禁用以上功能。

### 8.19.1 托盘能力

托盘菜单包含：

### 8.19.2 关闭窗口行为

用户可以选择：

- 关闭窗口时退出软件。
    
- 关闭窗口时最小化到系统托盘。
    

第一次关闭窗口时，可以询问用户偏好。

用户可以在设置中修改。

### 8.19.3 后台行为

托盘运行期间：

- 不执行高频目录扫描。
    
- 可以继续完成已经开始的 Git 状态刷新。
    
- 不自动执行 Fetch 或 Pull。

软件可以在启动后静默检查新版本。

用户可以关闭“启动时自动检查更新”。

### 8.22.2 更新提示

发现新版本后展示：

- 当前版本。
    
- 最新版本。
    
- 发布时间。
    
- 更新说明。
    

### 8.22.3 用户确认

软件不得自动下载或安装更新。

只有用户点击“下载并安装”后，才开始更新。

### 8.22.4 更新要求

更新过程必须保留：

- 项目列表。
    
- 扫描目录。
    
- 项目显示名称。
    
- IDE 配置。
    
- Pull 策略。
    
- 界面设置。
    
- 最近打开时间。
    

检查更新失败不得影响软件正常使用。

MVP 不支持：

- 强制更新。
    
- 测试版通道。
    
- 灰度发布。
    
- 多更新渠道。

## 8.23 安装版与免安装版

### 8.23.1 安装版

安装版支持：

- Windows 开始菜单快捷方式。
    
- 可选桌面快捷方式。
    
- 正常卸载。
    
- 软件更新。
    
- 配置保存在 Windows 当前用户数据目录。
    

### 8.23.2 免安装版

免安装版解压后直接运行。

配置、项目数据和错误日志保存在软件目录中。

建议目录结构：

```
LocalProjectManager/
├─ LocalProjectManager.exe
├─ data/
│  └─ app.db
├─ config/
│  └─ settings.json
├─ logs/
└─ updates/
```

### 8.23.3 免安装版权限

软件目录没有写入权限时：

- 停止写入配置。
    
- 明确提示用户将软件移动到可写目录。
    
- 不建议放置在 `C:\Program Files`。
    

### 8.23.4 数据隔离

安装版与免安装版的数据默认互不共享。

删除免安装版整个目录会删除软件配置，但不会删除任何被管理的 Git 项目。

# 12. 异常场景

## 12.1 扫描目录不存在

处理方式：

- 保留扫描目录配置。
    
- 标记目录不可用。
    
- 继续处理其他扫描目录。
    
- 提供重新选择或移除目录。
    

## 12.2 项目目录不存在

处理方式：

- 项目标记为“路径失效”。
    
- 保留项目记录。
    
- 禁用项目打开和 Git 操作。
    
- 提供重新指定路径。
    

## 12.3 Git 未安装

处理方式：

- 项目仍可导入和使用 IDE 打开。
    
- 禁用 Git 功能。
    
- 显示中文提示。
    

## 12.4 项目不是有效 Git 仓库

处理方式：

- 手动添加时阻止添加。
    
- 已导入项目失去 Git 元数据时标记异常。
    
- 不自动删除项目记录。
    

## 12.5 IDE 不可用

处理方式：

- 标记 IDE 不可用。
    
- 提示用户重新选择 IDE。
    
- 允许手动修复可执行文件路径。
    

## 12.6 网络不可用

处理方式：

- 本地项目管理功能正常工作。
    
- Fetch、Pull 和更新检查显示失败。
    
- 不影响项目打开和本地分支查看。
    

## 12.7 Git 认证失败

处理方式：

- 显示认证失败说明。
    
- 展示 Git 原始错误。
    
- 提供在终端中打开项目。
    
- 软件不要求用户输入密码或 Token。
    

## 12.8 Pull 冲突

处理方式：

- 停止 Pull。
    
- 标记项目存在冲突。
    
- 展示冲突文件数量。
    
- 引导用户通过 IDE 或终端处理。
    

## 12.9 数据保存失败

处理方式：

- 显示明确错误。
    
- 不覆盖仍然有效的旧数据。
    
- 提供打开日志目录。
    
- 免安装版提示检查软件目录写入权限。

# 13. 非功能需求

## 13.1 性能

建议 MVP 达到以下目标：

- 启动时优先展示缓存，不等待 Git 状态全部刷新。
    
- 搜索输入后快速返回结果。
    
- 扫描和 Git 刷新不得阻塞主界面。
    
- 后台 Git 状态刷新限制并发数量。
    
- 大型无关目录通过忽略规则提前跳过。
    
- 单个 Git 命令超时后不得长期占用任务线程。
    

## 13.2 稳定性

- 单个项目扫描失败不能中断全部扫描。
    
- 单个项目 Git 刷新失败不能影响其他项目。
    
- 配置和数据库写入应避免产生半写入数据。
    
- 软件异常退出后，下次启动仍能读取最近一次有效配置。
    
- 重复扫描不得创建重复项目记录。
    

## 13.3 易用性

- 所有危险或可能改变仓库状态的操作必须明确确认。
    
- Pull 和分支切换失败时提供可理解的中文说明。
    
- Git 原始错误可查看和复制。
    
- “从软件中移除”必须明确说明不会删除本地文件。
    
- 主要操作按钮名称应明确，不使用模糊图标代替关键文字。
    

## 13.4 兼容性

MVP 仅支持 Windows。

具体支持的最低 Windows 版本需要在技术选型阶段确定。

- 软件可以发现目录中的 Git 项目。
    
- 扫描结果默认全部选中。
    
- 用户可以取消部分项目。
    
- 嵌套仓库能够被识别和标记。
    
- 重复扫描不会产生重复项目。
    

## 15.2 项目列表

- 项目可以使用卡片视图和表格视图展示。
    
- 软件可以记住上次使用的视图。
    
- 项目按照最近打开时间排序。
    
- 用户可以通过名称和路径搜索项目。
    

## 15.3 IDE

- 软件可以检测 JetBrains 系列 IDE 和 VS Code。
    
- 用户可以手动添加其他 IDE。
    
- 项目第一次打开时要求选择 IDE。
    
- 后续可以使用上次选择的 IDE 直接打开。
    
- IDE 成功启动后更新最近打开时间。
    

## 15.4 Git 状态

- 可以展示当前分支。
    
- 可以展示未提交修改数量。
    
- 可以展示最近提交信息和时间。
    
- 可以展示远程仓库地址。
    
- 可以展示缓存的领先和落后数量。
    
- 启动时先展示缓存，再后台刷新。
    

## 15.5 Fetch

- 软件不会在后台自动 Fetch。
    
- 用户可以主动点击检查远程更新。
    
- Fetch 后更新远程分支和领先、落后数量。
    
- Fetch 失败时不影响本地项目管理。
    

## 15.6 分支切换

- 用户可以查看本地和远程分支。
    
- 用户可以搜索分支。
    
- 用户可以切换本地分支。
    
- 用户可以从远程分支创建本地跟踪分支。
    
- 存在本地修改时显示风险提示。
    
- 软件不自动 Stash。
    

## 15.7 Pull

- Pull 前展示确认窗口。
    
- 默认使用 `git pull --rebase`。
    
- 用户可以配置 Merge 或遵循 Git 设置。
    
- 用户可以设置全局策略和项目独立策略。
    
- 存在未提交修改时默认阻止 Pull。
    
- Pull 结果会保存为最近一次 Git 操作。
    

## 15.8 冲突

- Pull 冲突后项目会标记为“存在冲突”。
    
- 可以展示冲突文件数量。
    
- 可以使用 IDE或终端打开项目。
    
- 软件不在内部处理冲突。
    

## 15.9 路径失效

- 项目路径失效后记录不会自动删除。
    
- Git 和打开项目操作会被禁用。
    
- 用户可以重新指定路径。
    
- 用户可以只从软件中移除记录。
    

## 15.10 隐私

- 软件不上传项目数据。
    
- 软件不接入使用统计。
    
- 软件不自动上传错误日志。
    
- 只有 Fetch、Pull 和软件更新会按规则访问网络。

# 18. 产品一句话定义

本地 Git 项目管理器是一款面向 Windows 开发者的本地桌面工具，用于集中扫描和管理电脑中的 Git 项目，快速查看 Git 状态、切换分支、执行 Pull，并使用指定 IDE 一键打开项目。
